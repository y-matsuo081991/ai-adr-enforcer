# ADR 002: 非機能要件（冪等性と可用性）の担保戦略

* **ステータス:** Accepted
* **日付:** 2026-06-03

## 1. 背景と課題 (Context & Problem)
GitHub Action として動作する AI レビュアー (`ai-adr-enforcer`) の開発において、以下の2つの非機能要件（NFR）に関する運用上の課題が浮上しました。

1. **コメントのスパム化:** 開発者が Pull Request に新しくコミットを Push するたびに Action がトリガーされるため、AI が毎回新規にレビューコメントを投稿すると、PR のタイムラインが AI の発言で埋め尽くされてしまいます。
2. **AIのダウンタイムによる開発のブロック:** 外部の LLM API（Gemini等）は、レート制限や一時的なサーバーエラーのリスクを伴います。API のエラーによって Action が常に "Failed" (CI 失敗) になってしまうと、本来の人間の開発・マージ作業まで完全に停止してしまい、著しい開発体験 (DX) の悪化を招きます。

## 2. 決定事項 (Decision)

### 2.1 隠しマーカーを用いた Upsert 戦略 (Idempotency)
AI が PR にレビューコメントを投稿する際、コメント本文の末尾に、人間には見えないHTMLコメントの隠しマーカー（例: `<!-- ai-adr-enforcer-signature -->`）を必ず埋め込むアーキテクチャを採用します。
実行時はまず GitHub API (`listComments`) で既存のコメント一覧を取得し、このマーカーを持つコメントが存在すれば「新規投稿（Create）」ではなく、そのコメント ID を指定して「更新（Update）」を行います。これにより、何度 Action が実行されてもコメントが増殖しない冪等性（Idempotency）を担保します。

### 2.2 Fail-Open 設定によるフェイルセーフ (Resilience)
Action の `inputs` として、新たに `fail_open` (boolean) フラグを定義します。
Gemini API の呼び出し時などに例外（Error）が発生した場合、このフラグが `true` であれば `core.setFailed()` を呼び出さず、`core.warning()` でログを出力するだけに留めて Action を「正常終了（Pass）」として扱います。これにより、AIシステムに障害が起きても、人間の開発ブロック（CI の停止）を回避できる耐障害性（Fail-Open）を確保します。

## 3. もたらされる結果 (Consequences)
* **【ポジティブ】** PR のタイムラインが常に最新の1つの AI コメントのみに保たれ、レビューの視認性が向上します。
* **【ポジティブ】** 外部 API の障害がクリティカルパスにならず、アジャイルな開発プロセスを妨げません。
* **【ネガティブ】** Fail-Open を有効にした場合、「AIが落ちていたせいで、ADR違反のコードが気付かれずにマージされてしまう」リスク（False Negative）を受け入れることになります（これは運用時に利用者がフラグで選択可能なトレードオフとします）。
