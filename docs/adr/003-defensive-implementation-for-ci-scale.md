# ADR-003: AI監査機構における防御的実装とスケーラビリティの確保

* **ステータス:** Accepted (承認済)
* **日付:** 2026-06-04

## 📖 コンテキスト（背景と課題）
AI-Driven ADR Enforcerは、CIパイプライン上で動作する特性上、あらゆる規模や状況のPull Request（PR）に対して安定的かつコスト効率よく稼働する必要がある。
しかし、初期実装の全層監査（Full-Stack Audit）において、以下の構造的なスケーラビリティ課題と運用リスクが判明した。

1. **ペイロードの無制限な拡張（Token枯渇/コスト高騰）:**
   数万行に及ぶ巨大なリファクタリングPRや、自動生成ファイル（.lockファイル等）のDiffがそのままLLMに送信されると、Gemini APIのコンテキストウィンドウ上限を超過し、処理が必ず失敗する。また、従量課金におけるコストスパイクを引き起こす。
2. **外部依存のサイレント・ハングアップ（Resilienceの欠如）:**
   LLM API（Google GenAI）やGitHub APIへのHTTPリクエストにおいて明示的なタイムアウトが設定されていない場合、APIプロバイダ側の障害時にランナーが最大時間（例: 360分）ブロックされ続け、CIランナーの無駄な課金が発生する。
3. **フィードバックの冪等性（Idempotency）の欠如:**
   PRにPushが行われるたびに同じ違反コメントが連投されると、レビュアーおよび開発者のDX（開発体験）を著しく損なう。（※この課題については ADR-002 で Upsert 戦略を採用済みだが、本ADRでも重要要件として再定義する）

## 💡 決定事項 (Decision)
本アクションをエンタープライズの運用に耐えうる堅牢なシステムにするため、以下のアーキテクチャ制約を設ける。

1. **Diffサイズのハードリミットとノイズフィルタリング (Defensive Parsing)**
   * PRのDiff取得後、トークン消費に寄与しない自動生成ファイルや設定ファイル（例: `package-lock.json`, `*.svg`, `*.map`, `yarn.lock` など）はLLMへのプロンプト結合前に除外（Exclude）しなければならない。
   * フィルタリング後もDiffサイズが規定値（例: 100,000文字 または推定トークン上限）を超える場合は、APIへリクエストを送信せず、警告ログ（Warning）を出力した上で安全に監査をスキップ（Pass扱い）しなければならない。

2. **外部通信における厳格なタイムアウト設定 (Timeout Injection)**
   * Gemini APIおよびGitHub APIへのすべての非同期ネットワークリクエストには、必ず `AbortController` 等を用いた明示的なタイムアウト（例: 30,000ms）を設定しなければならない。
   * タイムアウト発生時は、既存の `fail_open` ロジックへ正しくルーティングさせ、CIを無期限にハングアップさせてはならない。

3. **ステートレスな冪等性の確保 (Idempotent Feedback)**
   * PRに対するコメント投稿処理（`postOrUpdateComment` 等）は、単なる POST（新規作成）ではなく、既存のBotコメントを検索し、存在する場合は内容を更新（PATCH または PUT）するUpsert処理でなければならない。

## 🎯 影響 (Consequences)

### ポジティブな影響 (Good)
* 巨大なPRや自動生成ファイルによるLLM API呼び出しエラーを未然に防ぎ、アクションの安定性（Pass/Failの予測可能性）が向上する。
* APIプロバイダの障害時にCIが長時間ブロックされる事態を防ぎ、インフラコストの無駄遣いを排除できる。
* コメントのUpsert化により、開発者のPRレビュー体験（DX）が向上する。

### ネガティブな影響・トレードオフ (Bad)
* 1回のPRで変更量がハードリミットを超えた場合、ADR違反がすり抜ける（False Negative）リスクを許容することになる。（※巨大なPR自体がアンチパターンであるため、チームの開発プロセスでPRを分割するよう促すことでカバーする）
* ネットワーク処理やDiff文字列操作のロジックが追加されるため、コードの複雑性がやや増す。
