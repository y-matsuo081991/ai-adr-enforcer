# ADR 010: Zodスキーマの整合性確保（suggestionのnullable許容）とタイムアウト・入力制限の最適化

## Status
* Proposed

## Context (背景と課題)
`ai-adr-enforcer@1.0.8` 以前において、実際のプロジェクト運用のCIパイプラインにおいて、PRのCIが「100%パースエラー」で落ちる、あるいは大規模なリファクタリングやコードの大量追加などの巨大なPRが発生した際にプロセスがハングアップ（タイムアウト）するという深刻な不具合が発生しました。
この根本原因は、以下の2つの設計・実装上のバグにあります。

### 1. ZodスキーマとLLM API（Structured Output）の不整合
LLM APIへのリクエストパラメータ（`responseSchema`）では、指摘事項がない（ADRに完全適合している）場合に `suggestion` フィールドに `null` を返すことを許可（`nullable: true`）していました。
しかし、取得したJSONを検証するクライアント側のZodスキーマ（`JudgeResultSchema`）では、`suggestion` が `z.string().optional()`（`undefined` は許容するが `null` は許容しない）と定義されていました。
このため、コードが完璧でADRを遵守している場合ほど、LLMが `suggestion: null` を返し、Zodによるバリデーションに引っかかって 100% `ZodValidationError`（パースエラー）で強制クラッシュしてしまうという矛盾した挙動が発生していました。

### 2. 巨大PR発生時のハングアップ（タイムアウト）
差分が極めて大きい巨大なPRが発生した場合、以下の要素が重なりデッドロック（タイムアウト）を引き起こしていました：
* デフォルトの `maxDiffSize` が `300,000` 文字と非常に大きく、極めて長大なDiffがそのままLLMに投入されてしまう。
* LLMは巨大なPRに対して膨大な評価（理由や複数の suggestion）を生成しようとし、最大出力トークン制限（Max Output Tokens）を突破。
* 結果として、不完全な（途中で途切れた）JSONが出力される。
* この場合、`JSON.parse` が `SyntaxError` を投げて即座に終了するはずですが、LLM API（`generateContent`）呼び出しにクライアントサイドのタイムアウトが設定されていませんでした。そのため、通信や推論が停滞している間、GitHub Actionsのプロセスが永遠にレスポンスを待ち続け（ハングアップ）、最終的にジョブ全体がタイムアウト（デッドロック）を誘発していました。

## Decision (決定事項)
本プロジェクトは、NDA（機密保持）に完全に適合するよう（ADR 001に則り）固有名詞や特定の企業ドメイン知識を排除しつつ、かつ堅牢で信頼性の高いアーキテクチャ・ガバナンス（Architecture Governance as Code）を実現するため、以下の3つの防衛的設計・修正を適用します。

1. **Zodスキーマの整合性確保 (`suggestion` の `nullable` 許容)**
   Zodのスキーマ定義（`JudgeResultSchema`）を `z.string().nullable().optional()` に修正し、LLMが指摘なしと判断して `null` を返した場合も正常に検証を通過できるようにします。
2. **LLM API呼び出しのクライアントサイドタイムアウト設定**
   `@google/genai` の `generateContent` 呼び出しにおいて、`config` に `httpOptions: { timeout: 30000 }`（30秒タイムアウト）を明示的に設定します。これにより、予期せぬ推論遅延や通信停滞によってプロセスがハングアップするのを防ぎ、速やかにエラーハンドリング（および `fail_open` に基づくフェイルセーフ）へ制御を移します。
3. **デフォルト `maxDiffSize` の引き下げ（ガードレールの強化）**
   デフォルトの最大Diffサイズ（`maxDiffSize`）を `300,000` 文字から `100,000` 文字に引き下げます。これにより、LLMのトークン制限や出力制限に引っかかるような極端に巨大なPRの入力を早期に検知し、ADR 006に基づき Fail-Closed (デフォルト) もしくは Fail-Open として即座に安全に処理します。

## Consequences (結果・影響)
* **[Good] 正常系PRの安定動作 (100%パースエラーの解消):** ADRを正しく遵守しているPRに対して、不要なバリデーションエラーが発生しなくなり、CIが安定してグリーンになります。
* **[Good] ハングアップ・デッドロックの防止:** 巨大PRに対してタイムアウト（30秒）と入力サイズ制限（100,000文字）という2重のガードレールが敷かれるため、GitHub Actionsのジョブが永遠にハングし続けるような無駄なリソース消費を防ぎます。
* **[Good] 予測可能性の向上:** 巨大なPRは速やかに（30秒以内、または入力リミット超過の瞬間に）検知され、人間による手動レビューへの促し（またはbypassラベルの使用）が行われます。
