# ADR 005: AIの誤検知（False Positive）に対する脱出ハッチ（Escape Hatch）の提供

## Status
* Proposed

## Context (背景と課題)
本ツールはLLM-as-a-Judgeを用いてアーキテクチャの監査を行います。しかし、LLMは確率的モデルであるため、100%の精度は保証できず、時として正当なコードを「規約違反」と誤検知（False Positive）する可能性があります。
現状の実装では、AIが `Fail` と判定すると `core.setFailed()` でCIが落ちます。リポジトリの設定でこのActionが「必須のステータスチェック（Required Status Check）」に指定されている場合、**人間が承認してもマージできないデッドロック状態（Single Point of Failure）** に陥るリスクがあります。

## Decision (決定事項)
開発チームがAIの判定を任意にオーバーライドできる **「Workflow-Level Bypass（ラベルによる脱出ハッチ）」** を実装します。

*   Pull Request に特定のラベル（例: `bypass-adr` または `override-adr`）が付与されている場合、本ツールのコアロジック（ADR読み込み、Diff取得、LLM推論）をすべてスキップし、即座に `Pass`（正常終了）として終了します。
*   アクション開始時の最初（Early Return）でこの判定を行います。

## Consequences (結果・影響)
*   **[Good] 開発アジリティの担保:** 誤検知発生時でも、開発者または管理者がラベルを付与するだけで即座にCIのブロックを解除でき、開発の停滞を防ぎます。
*   **[Good] 監査証跡（Audit Trail）の確保:** GitHubのPRタイムラインに「誰がラベルを付与してバイパスしたか」が記録されるため、セキュリティ監査の要件を満たします。
*   **[Bad] 運用の形骸化リスク:** ラベル付与の権限を適切に管理しないと、開発者が面倒なレビューを避けるために日常的にラベルを使用し、ADR監査自体が形骸化する恐れがあります（運用ルールの啓蒙が必要）。
