# ADR 006: 巨大PR（Size Limit超過）における Fail-Closed を基本とした防衛的設計

## Status
* Proposed

## Context (背景と課題)
GitHub Action として動作する本ツールは、1つのPull Request（PR）のDiff（差分コード）全体をコンテキストとしてLLMに送信します。
しかし、大規模なリファクタリングや自動生成ファイルのコミットによりDiffのサイズが極端に大きくなった場合、以下の問題が発生します。
1.  **トークン枯渇 (Token Exhaustion):** LLMのコンテキストウィンドウをオーバーし、APIがエラーを返す。
2.  **コスト高騰:** 大量トークン消費による想定外のAPI利用料の発生。
3.  **Bypass攻撃 (Payload Padding):** 悪意ある開発者が意図的に巨大なダミーファイルを含めることで制限を超えさせ、背後に仕込んだ違反コードをノーチェックで通過（Fail-Open）させるリスク。

現在の実装では、Diffが `DIFF_SIZE_LIMIT` (100,000文字) を超えた場合、警告ログを出力しつつ処理をスキップし、CIステータスは `Pass` とする（Fail-Open）暫定処置がとられています。

## Decision (決定事項)
巨大Diff処理時の基本ポリシーを **Fail-Closed（ブロック・要手動レビュー）** に変更し、以下の設計を適用します。

1.  **Fail-Closedの原則:** `DIFF_SIZE_LIMIT` を超過した場合、自動監査は不可能と判断し、**原則として `core.setFailed()` を呼び出してCIを落とします（Fail-Closed）。** これにより、Bypass攻撃を防ぎ、人間によるマニュアルレビューを強制します。
2.  **既存 `fail_open` オプションへの連動:** ユーザーがAPI障害時のフェイルセーフとしてアクションの引数に `fail_open: 'true'` を指定している場合にのみ、このSize Limit超過時も一貫して `Pass` とし、警告のみを出力する挙動とします。

## Consequences (結果・影響)
*   **[Good] セキュリティ強化:** 巨大PRを利用した自動レビューの意図的な回避（Payload Padding）を防ぐことができます。
*   **[Good] 運用の一貫性:** API障害時とリソース超過時のフォールバック挙動が `fail_open` オプション一つで統合されます。
*   **[Bad] False Positives (摩擦の増加):** 大規模な自動生成コード（ライブラリのアップデートなど）を含んだ正当なPRであっても、監査不能としてCIが落ちるようになり、前述の「脱出ハッチ（ADR 005）」等による開発者の手作業が必要になります。
